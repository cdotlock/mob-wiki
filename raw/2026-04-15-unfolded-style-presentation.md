# 当前展示方式说明：Unfolded 风格互动视觉小说原型

## 文档目的

这份文档描述当前 `No Rules` 原型在前端播放器中的实际展示方式。它不是在定义理想中的最终产品，而是在总结“现在这个原型已经如何展示内容”。

这里说的“类似 Unfolded”，指的是一种更接近移动端沉浸式互动小说的呈现逻辑：

- 以竖屏全屏阅读为核心
- 用整屏背景图承载场景氛围
- 用透明底角色立绘做人物出场与情绪切换
- 用悬浮式对话气泡承载文本，而不是 Episode 那种强舞台调度感的站位系统
- 用点击推进节奏，用少量分支选择打断线性阅读

当前原型的核心目标不是做“可自由操控的角色舞台演出”，而是做“移动端可顺滑阅读的互动剧本播放器”。

---

## 一、整体展示范式

### 1. 产品形态

当前原型本质上是一个移动端单页互动阅读播放器。

- 运行载体是一个 HTML 页面：`ep1-player.html`
- 展示内容是 Episode 1 的一段完整互动体验
- 页面在视觉上模拟手机原生阅读体验，而不是桌面网页布局
- 用户的核心操作只有两种：
  - 点击屏幕继续阅读
  - 在出现选项时点击按钮做选择

### 2. 与 Episode 风格的根本差异

当前版本虽然仍然使用“背景 + 角色立绘 + 对话 + 选择”的经典视觉小说结构，但展示逻辑更偏“阅读流”和“镜头流”，而不是“舞台流”。

它与传统 Episode 式展示相比，有几个明显差异：

- 场景是主画面，角色是叠加层，而不是一个强调角色站位编排的舞台。
- 对话框悬浮在画面上方的中下部，不是强占底部的大面积 UI 面板。
- 同一时刻通常只强调一个角色，而不是长期维持多角色同屏对话。
- 镜头语言依赖背景图切换、淡入、轻微平移和角色滑入，而不是复杂的角色 blocking。
- “内心独白”“旁白”“短信界面”被视为同等级叙事容器，和常规对白并列存在。

从体验上说，当前版本更像是在看一部会被你点击打断、并且偶尔需要你做决定的互动漫画/互动小说。

## 二、当前页面的舞台结构

### 1. 画布规格

播放器使用一个固定上限宽度的竖屏容器作为主舞台：

- 最大宽度：`393px`
- 高度：`100dvh`
- 容器居中
- 背景为黑色
- 整个容器都可点击，用于推进剧情

这个设计意味着：

- 在手机上，它基本会接近原生全屏内容区
- 在桌面上，它会保持一个窄幅、手机比例的阅读窗口
- 设计默认是“先服务竖屏移动端，再兼容桌面预览”

### 2. 分层结构

当前画面由多个绝对定位层叠加而成，从后到前大致如下：

1. `bg-back`
2. `bg-front`
3. 左侧角色槽 `char-left`
4. 右侧角色槽 `char-right`
5. 对话区 `dialogue-area`
6. 选项区 `choice-area`
7. 手机短信弹层 `phone-overlay`
8. 结束页 `end-screen`
9. 点击提示 `tap-indicator`

这套结构非常重要，因为它说明当前展示不是“一个统一的渲染组件”，而是“背景层 + 角色层 + 叙事 UI 层”的组合。

### 3. 当前镜头语言的基本原则

虽然页面里有左右两个角色槽，但当前引擎实际上遵循的是“单主角聚焦”原则：

- 场景切换时，先把人物全部隐藏，再换背景
- 角色对白时，确保只保留当前说话角色一侧的人物
- 主角内心戏时，优先确保 Malia 在场
- 当新角色出现时，引擎会让玩家点一次，再继续下一句文本

这意味着当前原型并不追求复杂多人同屏调度，而是追求“每一步都让玩家知道此刻该看谁”。

这是当前版本最像 Unfolded 风格的一点：阅读焦点始终单一、明确、无噪音。

## 三、素材如何被展示

### 1. 背景图的展示方式

当前背景素材来自 `assets/v2_scenes/`，例如：

- `bg_malias_bedroom.png`
- `bg_school_front.png`
- `bg_school_hallway.png`
- `bg_school_classroom.png`
- `bg_school_cafeteria.png`
- `bg_school_gymnasium.png`

背景图在页面中的处理方式如下：

- 使用 `background-image` 作为整层背景，而不是 `<img>`
- `background-size: cover`
- `background-position: center`
- 占满整个竖屏容器

这意味着素材会被当作“镜头底板”使用，而不是被精确按原尺寸展示。只要图足够大，系统会自动裁切，以保证画面始终充满整个屏幕。

### 2. 背景切换动画

当前背景切换是一个“旧图淡出 + 新图淡入 + 轻微横向平移”的组合：

- 旧背景被保留在 `bg-back`
- 新背景加载到 `bg-front`
- 新背景初始位置从 `left center` 开始
- 在 `1.2s` 内缓慢移动到 `center center`
- 同时做透明度淡入

这个处理非常关键，因为它让静态图获得了一点“镜头推移”的感觉。

它不是严格意义上的摄像机运动，但足够制造一种更电影化、比纯静态切图更高级的阅读节奏。

### 3. 角色立绘的展示方式

当前角色素材来自 `assets/v2_cutout/`，全部是透明背景 PNG。

角色命名遵循“人物 + 集数/版本 + 表情动作”的方式，例如：

- `malia_ep1_neutral_phone_cutout.png`
- `mauricio_ep1_neutral_smirk_cutout.png`
- `easton_ep1_neutral_vulnerable_cutout.png`
- `mark_ep1_neutral_grin_cutout.png`
- `elias_ep1_neutral_calm_cutout.png`

当前角色图的展示规则如下：

- 角色槽绝对定位在屏幕底部
- `bottom: -45%`
- `height: 130%`
- 图片高度撑满角色槽，宽度自适应

这组参数的实际效果是：

- 角色并不是完整地“站在屏幕里”
- 角色会被有意放大
- 下半身部分被压到屏幕外
- 视觉焦点更集中在上半身、脸部、姿态和气质

这非常符合当前项目的目标，因为它强调的是情绪与阅读，而不是全身动作表演。

### 4. 角色进出场动画

角色不是直接出现，而是从画面边缘滑入：

- 左侧角色从左边进入
- 右侧角色从右边进入
- 滑入时长约 `0.4s`
- 滑出时长约 `0.3s`

这会产生一种很轻量的“角色走进镜头”的感觉。

当前版本没有做：

- 呼吸动画
- 待机循环动画
- 局部缩放
- 立绘抖动
- 镜头推近角色脸部

所以它的动感来自“少量但稳定的转场”，而不是高频动态特效。

### 5. 表情切换方式

角色表情切换目前是直接换图，不做中间过渡：

- 当前角色已经在场时，只更新 `img.src`
- 不额外播放淡入淡出
- 表情变化立即生效

这使得角色反应显得很直接，节奏也更利落。

不过当前素材层有一个现实情况：

- 并不是每个表达都对应独立立绘
- 有些脚本中的情绪标签会映射到同一张图

例如某些 `neutral_*` 变体，本质上只是复用了相近表情的 PNG。

这说明当前展示系统已经支持“细粒度表情命名”，但素材资产本身还没有完全一一补齐。

## 四、当前 UI 的叙事容器

当前播放器不是只有一种“对话框”。它实际上有 5 种叙事容器。

### 1. 常规对白容器

常规对白用于角色之间的说话内容。

样式特征：

- 出现在屏幕大约 `62%` 高度的位置
- 宽度 `317px`
- 说话人姓名显示在一个彩色圆角 name tag 上
- 文本显示在白色对话气泡内
- 气泡有黑色描边与阴影

布局规则：

- Malia 的对白通常左对齐
- 其他主要角色通常右对齐
- 无立绘角色也走左对齐逻辑

这种“左右分区”的呈现，不是在模拟聊天室，而是在强化“谁掌握当前叙事主动权”。

### 2. 主角内心独白容器

`YOU` 类型文本不是普通对白，而是主角的内心活动。

它的样式和常规对白明显不同：

- 上方有一个 thinking bubble 图标
- 文本仍在白色气泡内
- 气泡圆角更大，更像思考泡泡而不是说话泡泡
- 默认左对齐

这类内容是当前原型非常重要的组成部分。

它让玩家持续站在 Malia 的主观视角里，而不是被动旁观剧情。

### 3. 旁白容器

旁白与对白、内心戏又不同。

当前样式是：

- 气泡居中
- 没有角色名
- 白色胶囊型文本块
- 文字居中排版

旁白的作用不是表达角色说话，而是承担镜头切换、时间变化、场景说明和情绪总结。

在当前播放器里，旁白是非常频繁使用的一种叙事容器，它承担了大量“像漫画分镜字幕一样”的工作。

### 4. 手机短信弹层

短信不是简单地用对白模拟，而是单独弹出一个手机界面：

- 整屏加黑色半透明遮罩
- 中间弹出一个 iMessage 风格卡片
- 顶部显示联系人
- 中间显示消息气泡
- 底部显示输入栏占位

消息本身分为两种视觉方向：

- 对方消息：灰色气泡，左对齐
- 自己消息：蓝色气泡，右对齐

这类展示特别适合处理：

- ex 发来的消息
- 秘密短信
- 夜间私聊
- 身份揭示
- 推动悬念的短文本节点

在当前原型里，短信层是叙事节奏的重要断点，因为它会暂时中止普通阅读流，让用户进入一个更“贴身”的信息界面。

### 5. 选择容器

当剧情进入分支点时，常规对话框被隐藏，选项区直接取代对话区出现。

当前选项样式：

- 位于和对话框同一纵向区域
- 纵向排列
- 白底黑边
- 高度 `56px`
- 大圆角
- 重字重标题式按钮文案

交互行为：

- 玩家点击选项后，不再显示普通对话框
- 选项立即消失
- 系统记录分支名
- 剧情跳入对应 branch

这意味着当前版本的选择不是一个“额外弹窗”，而是阅读流本身的一部分。

## 五、当前交互节奏是如何推进的

### 1. 核心推进方式：Tap to Continue

整个原型的默认推进方式是点击屏幕继续。

页面底部常驻一个轻微脉冲的文案：

`Tap to continue`

它的作用不是提示复杂操作，而是把体验稳定为一种“节拍式阅读”：

- 读一段
- 点一下
- 再读一段
- 再点一下

这种节奏对互动小说非常重要，因为它比自动播放更可控，也比传统长文本滚动阅读更有镜头感。

### 2. 自动推进与手动推进的边界

当前引擎把脚本步骤分成两类：

第一类是自动推进步骤：

- 切背景 `bg`
- 显示角色 `show`
- 切表情 `expr`
- 隐藏角色 `hide`
- 程序等待 `wait`
- 分支控制 `branch_start / branch_end / converge`

第二类是必须等待玩家动作的步骤：

- 旁白 `narrator`
- 内心戏 `you`
- 对白 `dialogue`
- 短信 `phone`
- 选择 `choice`

这套设计的实际效果是：

- “镜头动作”会自己发生
- “文本内容”必须由玩家逐步消费

因此玩家感知到的是一个有镜头语言的阅读器，而不是一个需要自己操心调度的系统。

### 3. 新角色出场时的节奏控制

当前版本有一个非常明确的节拍设计：

- 如果新角色滑入画面
- 系统不会立刻继续跑下一句
- 而是停住，等待玩家点一次

这个小设计非常值钱，因为它让“角色出场”本身成为一个叙事事件。

它等于是在说：

- 先看人
- 再看他说什么

这比“角色和文字同时挤进屏幕”更有情绪停顿感。

### 4. 分支处理方式

分支逻辑目前是非常清晰的脚本式分支：

- 选择步骤记录 `chosenBranch`
- 播放器读到 `branch_start`
  - 如果 branch 名匹配，则继续执行
  - 如果不匹配，则一直跳到对应 `branch_end`
- 所有分支在 `converge` 后回到主线

这说明当前版本使用的是“局部分叉、快速回收”的轻分支结构，而不是大规模树状叙事。

这种结构很适合：

- 第一集试玩
- 早期角色好感差异化
- 不影响主线产能的互动体验

### 5. 音乐处理

当前播放器只加载了一首循环 BGM：

- 文件：`assets/music/euphoria.mp3`
- 循环播放
- 音量约 `0.3`

音乐不是自动开播，而是在用户第一次交互后启动。这是一个符合浏览器自动播放策略的处理方式。

这个细节虽然简单，但对体验很重要：

- 用户一进入页面时不会被浏览器拦截
- 一旦开始互动，气氛就会稳定建立起来

### 6. 结束方式

当脚本走到 `end` 时：

- 对话框隐藏
- 所有角色隐藏
- 显示一个全屏结束层
- 文案为 `End of Episode 1 / To be continued...`

这是一种非常标准的章节收尾方式：

- 不继续给额外操作
- 不弹复杂结算页
- 用留白感结束体验

## 六、脚本是如何驱动画面的

当前展示方式不是手写 DOM 流，而是“脚本数组驱动渲染”。

播放器内部存在一个 `SCRIPT` 数组。每一个对象就是一步叙事指令。引擎按顺序读取它们，并把它们翻译成画面变化。

### 1. 当前脚本步骤类型

当前原型已经支持以下步骤类型：

| 类型 | 作用 | 是否需要玩家动作 |
| --- | --- | --- |
| `bg` | 切换背景场景 | 否 |
| `show` | 让角色入场 | 新角色入场后会停住等待 |
| `hide` | 让角色退场 | 否 |
| `expr` | 切换表情/姿态 | 否 |
| `narrator` | 显示旁白 | 是 |
| `you` | 显示主角内心独白 | 是 |
| `dialogue` | 显示角色对白 | 是 |
| `phone` | 显示短信卡片 | 是 |
| `choice` | 显示选项 | 是 |
| `wait` | 程序延时 | 否 |
| `branch_start` | 分支开始 | 否 |
| `branch_end` | 分支结束 | 否 |
| `converge` | 分支汇合 | 否 |
| `end` | 结束章节 | 否 |

### 2. 从脚本到展示的典型映射

一个典型片段会这样运作：

1. `bg`
2. `show`
3. `narrator`
4. `dialogue`
5. `expr`
6. `you`
7. `choice`

翻译成玩家看到的体验就是：

1. 镜头切到新场景
2. 某个角色滑入
3. 出现一段旁白说明
4. 角色开始说话
5. 角色表情切换
6. 主角给出内心反应
7. 玩家被要求选一个态度或行动

这是一种非常稳定的互动小说节奏模板，后续扩内容时复用成本很低。

### 3. 当前脚本与素材的关系

脚本里的场景名和角色表情名不是直接展示的，它们还要经过两层映射：

- 场景名 -> `SCENES`
- 角色状态名 -> `SPRITES`

例如：

- `malias_bedroom_morning` 映射到 `assets/v2_scenes/bg_malias_bedroom.png`
- `MAURICIO + neutral_smirk` 映射到某张 `v2_cutout` PNG

这层映射很有价值，因为它让脚本作者不需要关心真实文件名，只需要写叙事语义。

也就是说，当前系统已经在往“内容脚本层”和“素材资源层”解耦的方向走。

## 七、Episode 1 当前是如何被展示出来的

为了避免文档只停留在抽象层，下面用当前 `ep1-player.html` 里的实际内容，说明一遍用户进入页面后会经历什么。

### 1. 开场：卧室 + 手机消息 + 主角内心戏

用户进入页面后，首先看到的是：

- Malia 卧室背景
- Malia 的 phone 姿态 cutout 在左侧出现
- 一段连续的旁白
- 几句主角内心独白
- Easton 发来的短信弹层

这里的展示重点不是信息量，而是建立体验基调：

- 先用静态画面让用户进入角色空间
- 再通过短信界面把“旧关系未断”的冲突直接丢出来
- 最后用大量内心戏把玩家锁进 Malia 的主观视角

### 2. 第一段外部世界：学校门口 + Josie 出场

镜头切到学校门口后：

- 背景换成学校外景
- Malia 重新出现
- Josie 作为右侧角色滑入
- 对话框开始承担更轻快、更生活化的节奏

这一步的作用是把用户从“内心戏模式”切换到“社交场景模式”，同时验证当前系统支持角色对白驱动的轻喜剧节奏。

### 3. 第二段张力建立：走廊 + Mauricio 出场

接着镜头切到学校走廊：

- 仍然保留 Malia 的主观视角
- Josie 和 Malia 先铺一段“别看他”的过门对白
- Mauricio 再作为一个事件性角色入场

这里最重要的不是他说了什么，而是“他出现了”本身。

因为当前引擎会在角色新入场时停住等待点击，所以 Mauricio 的出现被做成了一个节拍点，而不是普通信息更新。

### 4. 中段切换：课堂与食堂

课堂段落里，系统展示了两件事：

- 无立绘角色也能承载对白，比如老师
- 内心独白和角色出场可以交替组成阅读节奏

食堂段落里，Mark 登场后，展示风格明显又变了一次：

- 文本更快
- 气质更外放
- 对白推动更强
- 节奏更接近 rom-com

这说明当前容器系统并不只是“能显示字”，而是能通过容器、站位和角色切换去承接不同关系氛围。

### 5. 第一个互动点：Easton / Mark 二选一

在食堂段落后，页面第一次进入真正意义上的互动节点：

- 常规对话框消失
- 两个大按钮出现
- 用户在“让 Easton 过来”和“让 Mark 制造混乱”之间做选择

这个节点很典型，因为它展示了当前系统的分支哲学：

- 不是让玩家重写命运
- 而是让玩家决定“这一刻如何回应关系压力”

这和 slow-burn romance 的需求是匹配的。

### 6. 分支段落：局部分叉，但很快回主线

如果玩家让 Easton 过来：

- 会进入一段面对面紧张对话
- Easton 作为右侧角色和 Malia 形成正面关系场
- 文本重点是未解决的旧情

如果玩家让 Mark 出手：

- 会进入一个更轻松、更回避冲突的段落
- 背景切走
- 用幽默方式躲开正面接触

虽然两个分支的情绪不同，但它们都很快收束并重新汇回主线。

这说明当前系统的目标不是做复杂树状剧情，而是在不炸内容成本的前提下，给用户“我的态度不一样”的参与感。

### 7. 后段铺线：体育馆 + Elias + Mauricio

汇流之后，镜头进入体育馆：

- 背景切到 gym
- Elias 和 Mauricio 依次进入视野
- Josie 用对白继续强化 Malia 的感知和否认

这一段的作用，不是完成某条路线，而是同时给多个可攻略对象继续埋钩子。

从展示上看，它证明了当前系统很适合“角色串联式登场”：

- 同一个场景里
- 用一个个简短的出场与注视
- 累积关系张力

### 8. 收尾：夜晚卧室 + 对窗 + 神秘短信

最后镜头回到卧室：

- 氛围明显从白天社交切回夜晚私密
- 文本重心从对白切回内心戏与旁白
- 再通过一条“nice curtains, Butterfly”的短信做结尾

这是当前展示系统一个非常强的能力点：

- 白天段落适合用对白和角色出场推动
- 夜晚段落适合用背景、独白、短信承接悬念

这两种模式在同一播放器里可以平滑切换。

### 9. 章节结尾的体验感受

最后所有内容结束后：

- 角色退场
- 对话收起
- 只留下 End Screen

这会让玩家感知到一种非常明确的“第一集结束了”的章节边界，而不是网页内容播完了。

## 八、为什么当前展示会让人感觉像 Unfolded

如果要用一句话概括当前展示气质，可以说：

它把互动内容当成“沉浸式阅读界面”来呈现，而不是“角色表演引擎”。

具体体现在以下几点。

### 1. 画面重心在阅读，而不是操作

玩家绝大多数时候只需要点击，不需要管理复杂系统。

这让体验重点回到：

- 看图
- 读字
- 感受关系变化
- 在少数关键点做决定

### 2. 场景图是情绪底座

背景图始终铺满整个屏幕，并带有轻微平移。这使场景不是“说明性插图”，而是“阅读情绪空间”。

### 3. 角色以情绪肖像的方式存在

立绘被放大、底部裁切、强调上半身，这让人物更像情绪肖像，而不是可被调度的舞台演员。

### 4. 文本容器不是单一的

对白、旁白、内心戏、短信、选项都拥有独立容器。这种容器切换本身就构成了阅读节奏。

### 5. 节奏由 UI 切换和镜头切换共同塑造

当前“好看”的地方不来自复杂动画，而来自这些轻量元素的组合：

- 背景淡入
- 角色滑入
- 不同文本容器切换
- 选项插入
- 短信弹层打断

这套组合非常适合互动小说，而不适合传统角色演出式游戏。

---

## 九、当前版本的能力边界与限制

这份文档不只要描述优点，也要把“现在还没做到什么”说清楚。

### 1. 实际上仍然是单角色焦点系统

虽然页面有左右两个角色槽，但当前对白阶段通常只保留一个主要角色。多人同屏对话并不是当前原型的重点能力。

### 2. 表情与动作资产还不够丰富

脚本层已经支持很多情绪标签，但素材层并没有为每个标签准备真正独立的立绘，部分标签仍在复用。

### 3. 场景数量有限

当前场景主要覆盖：

- Malia 卧室
- 学校门口
- 学校走廊
- 教室
- 食堂
- 体育馆

对一个完整赛季式互动内容来说，这只是第一批基础场景。

### 4. 时间变化还没有被视觉化区分

像 `morning / noon / afternoon / evening` 这样的脚本语义，目前大多还是复用同一张背景图，而不是有完全独立的光照版本。

### 5. 缺少更复杂的演出功能

当前版本还没有：

- 镜头缩放
- 人物局部特写
- 多角色分层深度
- 角色震动或冲击特效
- 打字机效果
- 历史回看
- 自动播放
- 存档/读档
- 章节选择
- 好感数值反馈 UI

因此它现在更接近一个“高保真原型播放器”，而不是完整商业化客户端。

## 十、如果要一句话定义当前展示系统

当前系统可以被定义为：

**一个面向竖屏移动端的脚本驱动互动小说播放器，用整屏背景、放大角色 cutout、悬浮式文本容器、点击推进和轻分支结构，去实现一种更接近 Unfolded 的沉浸式阅读体验。**

## 十一、给产品、设计、前端的共识结论

### 产品视角

当前展示方式已经证明，这套内容不需要 Episode 那种重舞台系统，也可以成立。它依赖的是阅读节奏、情绪图像和容器切换，而不是复杂角色调度。

### 设计视角

当前视觉重点不是“UI 做满”，而是“把 UI 做轻，让素材和文本成为主角”。背景、立绘、name tag、对话气泡、短信卡片，这 5 个元素已经构成了完整视觉语言。

### 前端视角

当前结构已经具备一个可扩展引擎的雏形：

- 脚本层和素材层是分开的
- 步骤类型已经比较完整
- 分支处理已经存在
- 后续可以继续往动画、音效、状态系统、商业化 UI 上扩

### 内容视角

当前展示方式天然适合：

- 高情绪密度的青春校园题材
- 强主观视角的 romance 内容
- 需要频繁内心戏的 slow-burn 叙事
- 以“关系张力”而不是“大动作场面”为驱动力的剧情

## 附：当前原型中最核心的展示规则

为了方便后续统一理解，可以把当前系统压缩成以下 10 条规则：

1. 全部内容默认按竖屏移动端展示。
2. 背景图铺满全屏，并通过淡入和平移制造镜头感。
3. 角色使用透明底放大 cutout，强调情绪而不是全身动作。
4. 新角色出场要滑入，并给用户一个停顿。
5. 同一时刻通常只强调一个角色。
6. 对白、旁白、内心戏、短信、选项是 5 种独立叙事容器。
7. 玩家默认通过点击推进，而不是滚动阅读。
8. 选择点是局部分叉，之后快速汇回主线。
9. 音乐在第一次交互后开始，维持沉浸感。
10. 整个系统的核心不是“表演”，而是“可视化阅读”。
